Предложение по развитию архитектуры
Проект: Proposal
Статус документа: анализ и рекомендуемое решение, без начала реализации
Дата: 2026-04-22

1. Краткий вывод

Текущая система уже является микросервисной, но она остается файлово-центричной и ориентированной на умеренные объемы данных:
- backend выступает как API gateway и orchestration layer;
- асинхронность построена на Celery + Redis;
- raw / archive / normalized / model artifacts хранятся в файловой системе;
- обучение и автообучение работают через локальные временные директории и промежуточные CSV;
- схема БД инициализируется через SQLAlchemy `create_all()`, а не через управляемые миграции.

Для требований:
- до 50 000 архивов и файлов;
- до 200 ГБ общего объема;
- поэтапное внедрение без поломки текущей системы;
- RabbitMQ как надежный брокер;
- временная БД после разархивации;
- хранение нормализованных данных в БД;
- ручная очистка staging и normalized данных;
- обязательное версионирование моделей;

я не рекомендую "большой переписывательский" подход. Правильный путь: эволюционная миграция текущей архитектуры в событийно-ориентированную, с выделением staging/normalized data layers и переводом межсервисного обмена на RabbitMQ.

2. Что показал анализ текущего кода

2.1. Сильные стороны текущей системы

- Есть понятное разделение сервисов: backend, normalization-data, training-service, model-analiz, log-service.
- Уже есть асинхронный execution model через Celery.
- Есть доменные сущности для datasets, feature schemas, training runs, model artifacts, auto-training archives/jobs.
- Есть chunk upload для архивов и данных.
- Есть базовая наблюдаемость через structured logging и task tracking.

2.2. Ключевые ограничения текущей реализации

1. Очереди и асинхронность
- Сейчас Celery использует Redis как broker и result backend.
- В `backend/app/services/tasks.py` задача отправляется в брокер напрямую через `celery_app.send_task(...)`, а затем отдельно пишется запись в БД.
- Это означает отсутствие transactional outbox и отсутствие гарантии "DB commit + message publish" как единой операции.
- При росте нагрузки это создает риск потери сообщений, дублей, гонок и состояния "в БД запись есть, а сообщение не доставлено" или наоборот.

2. Хранилище данных
- Архивы, сырье, нормализованные датасеты и модели сейчас лежат в `app-data/*`.
- `training_service/app/worker.py` при автообучении:
  - разархивирует архивы на диск;
  - нормализует файлы в CSV на диск;
  - читает нормализованные CSV в pandas DataFrame;
  - объединяет их через `pd.concat(...)`;
  - затем сохраняет общий normalized CSV.
- Такой подход не рассчитан на 50 000 архивов / 200 ГБ и упрется в disk I/O, память, длительные временные окна и неустойчивость при рестартах.

3. Отсутствие staging data layer
- После разархивации данные не попадают во временную БД.
- Система работает по схеме "архив -> файлы на диске -> нормализация -> CSV на диске".
- Для больших объемов данных это слабое место по управляемости, очистке, дедупликации и повторной обработке.

4. Отсутствие normalized data layer в БД
- Нормализованные данные сейчас сохраняются как файлы.
- Это усложняет:
  - версионирование входа для обучения;
  - ручную очистку по доменным правилам;
  - повторное обучение на snapshot dataset;
  - аудит происхождения данных;
  - выборочное переиспользование подготовленных данных.

5. Слабое версионирование моделей
- В `ModelArtifact` есть lifecycle status (`candidate/promoted/deprecated`), но нет обязательного version field и registry model.
- Сейчас promotion работает по `model_type`, а не по модельному реестру и версии.
- Нет явной связи "версия модели -> версия датасета -> версия feature schema -> версия кода -> версия гиперпараметров".

6. Управление схемой БД
- В `backend/app/db/init_db.py` используется `Base.metadata.create_all(...)`.
- Это подходит для локального старта, но не подходит для управляемой эволюции production-схемы.
- Для поэтапной миграции нужны Alembic migrations.

7. Конфигурация
- Конфигурация уже в целом сведена в `.env` и `.env.example`, что хорошо.
- Но `.env` и `.env.example` уже расходятся по составу переменных.
- Для дальнейшей эволюции нужен единый inventory конфигурации и strict sync между `.env.example` и реальным `.env`.

3. Рекомендуемое архитектурное решение

3.1. Базовый принцип

Сохраняем текущую систему как основу и вводим новые уровни по этапам:
- RabbitMQ как брокер команд и событий;
- staging data layer для временных распакованных и разобранных данных;
- normalized data layer как источник истины для обучения;
- model registry с обязательными версиями;
- cleanup subsystem;
- миграции схемы БД;
- сохранение совместимости текущих API и UI по максимуму.

3.2. Целевая логическая схема

Рекомендую следующую целевую структуру:

1. Core operational layer
- текущий backend;
- текущие справочники, пользователи, задания, метаданные архивов, datasets, task records.

2. Messaging layer
- RabbitMQ как основной message broker;
- Redis можно сохранить временно как result backend Celery и/или cache layer;
- публикация сообщений только через outbox pattern.

3. Staging data layer
- временное хранилище для данных после разархивации и первичного разбора;
- данные хранятся chunk/partition-based;
- после успешной нормализации staging-данные удаляются вручную или по retention policy.

4. Normalized data layer
- нормализованные данные сохраняются в БД;
- они становятся каноническим источником данных для обучения;
- исходные разархивированные файлы после успешной загрузки и нормализации подлежат удалению.

5. Training and model registry layer
- обучение читает данные из normalized layer;
- модели пишутся в registry с обязательной версией, lineage и статусом.

6. Cleanup and retention layer
- ручная очистка staging и normalized данных;
- безопасная очистка артефактов и временных файлов;
- dry-run preview перед destructive operations.

4. RabbitMQ: как внедрять без поломки

4.1. Что я рекомендую

Не заменять все одним махом. Правильный первый шаг:
- оставить Celery как execution framework;
- перевести Celery broker с Redis на RabbitMQ;
- Redis оставить как result backend на переходный период;
- в backend добавить transactional outbox;
- доменные команды и события передавать через RabbitMQ.

4.2. Почему именно так

Плюсы:
- минимально инвазивно для текущего кода;
- не ломает текущую модель Celery workers;
- повышает надежность доставки;
- дает масштабируемую routing model через exchanges/queues;
- позволяет постепенно отделить orchestration commands от worker execution.

4.3. Что должно появиться

Новые logical message categories:
- commands:
  - archive.extract.requested
  - staging.load.requested
  - normalization.requested
  - training.requested
  - cleanup.requested
- events:
  - archive.upload.completed
  - archive.extracted
  - staging.load.completed
  - normalization.completed
  - training.completed
  - model.version.created
  - cleanup.completed
  - job.failed

Технические требования:
- dead-letter queues;
- retry policy с ограничением числа повторов;
- correlation_id;
- idempotency_key;
- message versioning;
- consumer idempotency;
- outbox publisher.

5. Временная БД после разархивации

5.1. Мое рекомендуемое решение

На первом этапе не вводить сразу отдельный новый DB-engine. Чтобы не ломать текущую систему, staging data layer лучше начать как отдельную логическую область внутри текущего PostgreSQL-кластера:
- либо отдельная схема `staging`,
- либо отдельная БД в том же PostgreSQL контейнере.

Мой рекомендуемый путь:
- логически проектировать как отдельную БД `staging_db`;
- физически на первом этапе можно развернуть это в том же PostgreSQL instance;
- позже, если объемы и нагрузка вырастут, вынести staging_db на отдельный инстанс.

5.2. Что туда писать

После разархивации данные не должны бесконтрольно жить на файловой системе. Правильный поток:

архив -> extraction worker -> parser/adapter -> staging tables

В staging layer должны храниться:
- metadata архива;
- metadata extracted files;
- file checksum;
- parser format;
- ingestion batch id;
- chunk index;
- row count;
- parse errors;
- raw structured chunks в табличном виде;
- expiration/retention markers.

5.3. Почему staging DB нужна

- обеспечивает повторяемую обработку;
- отделяет ingestion от normalization;
- позволяет чистить сырые данные отдельно;
- уменьшает жизненный цикл временных файлов;
- дает checkpoint/restart semantics;
- позволяет обрабатывать 200 ГБ без требования держать весь pipeline в памяти.

6. Нормализованные данные в БД

6.1. Рекомендуемый подход

Нормализованные данные должны стать отдельным слоем данных, а не только CSV на диске.

Рекомендую сохранять:
- dataset snapshot metadata;
- partition metadata;
- row statistics;
- schema version;
- feature family set;
- label distribution;
- lineage;
- сами нормализованные записи chunk/partition based.

6.2. Важный engineering trade-off

Я не рекомендую обещать, что весь training будет всегда читать "строго из SQL rows в лоб" для всех алгоритмов. Для части ML-стека эффективнее:
- БД выступает источником истины;
- training worker читает данные из normalized layer батчами;
- при необходимости materializes временные локальные shards/parquet/csv только в рамках конкретного job execution;
- после завершения job временные файлы удаляются.

Такой подход лучше, чем пытаться обучать все модели напрямую из огромных SQL select без промежуточного batching layer.

6.3. Что удалять

После успешной записи в normalized layer:
- extracted raw files удаляются;
- staging raw payload можно удалять сразу или по короткому retention;
- archive можно хранить до завершения полной цепочки или по policy.

7. Очистка staging и normalized данных

Нужна обязательная функциональность ручной очистки.

Рекомендую ввести:
- API очистки staging datasets;
- API очистки normalized datasets;
- UI с подтверждением и dry-run preview;
- очистку по критериям:
  - старше N дней;
  - по dataset version;
  - по source_type;
  - по job id;
  - orphaned data;
  - failed pipelines;
- запрет удаления данных, на которые ссылаются активные training runs или promoted models.

Отдельно нужны:
- cleanup jobs table;
- retention policies;
- журнал удалений;
- soft validation перед hard delete.

8. Версионирование моделей: обязательное, а не факультативное

8.1. Текущее ограничение

Сейчас `ModelArtifact` недостаточен для полноценного model registry.

8.2. Что нужно добавить

Рекомендую ввести отдельный model registry слой:

- model_registry
  - registry_id
  - model_family
  - problem_type
  - source_type
  - owner

- model_version
  - version_id
  - registry_id
  - version_number
  - semantic_version или monotonic revision
  - training_run_id
  - normalized_dataset_snapshot_id
  - feature_schema_id
  - code_version
  - hyperparameters
  - metrics
  - artifact_checksum
  - created_at
  - stage: candidate / validated / promoted / deprecated / archived

- model_version_artifact
  - artifact path
  - format
  - checksum
  - size

8.3. Обязательные правила

- Нельзя создавать модель без версии.
- Нельзя promoted model без привязки к dataset snapshot и schema version.
- Нельзя перезаписывать существующую версию.
- Promotion должен работать на уровне `registry + version`, а не только `model_type`.

9. Поддержка 50 000 архивов и 200 ГБ

Чтобы такие объемы были реально достижимы, система должна перейти с file-centric bulk flow на partitioned asynchronous pipeline.

Обязательные технические меры:
- chunk upload и resumable upload уже есть, их нужно сохранить;
- pagination для списков архивов, datasets, jobs;
- отказ от пересканирования директорий на каждый запрос;
- запрет глобальных `pd.concat` на все входы в памяти;
- обработка архивов по batch/window;
- ограничение concurrency по worker type;
- backpressure через очереди RabbitMQ;
- checksum-based deduplication;
- partitioning staging/normalized tables;
- cleanup временных файлов сразу после успешной загрузки в БД;
- job checkpointing;
- retry-safe idempotent consumers.

10. Поддержка форматов

Требование "независимо от формата" я рекомендую трактовать не как буквальное "абсолютно любой формат", а как extensible adapter architecture.

Правильная модель:
- registry архивных форматов;
- registry парсеров исходных данных;
- registry нормализаторов;
- fallback policy для unsupported formats;
- пер-file diagnostics.

Это позволит безопасно расширять поддержку форматов без переписывания core pipeline.

12. Что нужно изменить в сервисах

12.1. Backend
- перестать напрямую публиковать задачи без outbox;
- стать orchestrator + control plane;
- добавить endpoints для cleanup, registry, staging/normalized monitoring;
- добавить pagination и filtering для объемных списков;
- добавить correlation_id propagation.

12.2. Archive ingestion / auto-training
- текущий upload flow сохранить;
- после `complete` публиковать событие, а не запускать длинный synchronous flow;
- metadata архива сразу фиксировать в core DB;
- дедупликация по checksum и relative path.

12.3. Extraction workers
- вынести extraction в отдельную стадию;
- писать результат в staging layer chunk-by-chunk;
- не хранить огромные распакованные деревья дольше необходимого.

12.4. Normalization workers
- читать staging partitions;
- писать normalized partitions;
- обновлять dataset snapshot metadata;
- удалять staging raw payload после success.

12.5. Training workers
- читать normalized dataset snapshots;
- не строить глобальный единый DataFrame на весь входной массив;
- использовать batch/partition processing;
- сохранять model version + lineage.

12.6. Model service
- читать только promoted model version;
- поддерживать явный registry/version lookup.

13. Конфигурация: только через .env и .env.example

Это требование правильное. Его нужно формализовать.

Обязательные правила:
- все runtime settings только через `.env`;
- `.env.example` должен быть полным и актуальным контрактом конфигурации;
- запрещены "скрытые" env variables, не отраженные в `.env.example`;
- должна быть проверка конфигурации на startup.

14. Порядок внедрения

Я рекомендую следующий план.

Этап 0. Стабилизация основы
- инвентаризация конфигурации;
- приведение `.env` и `.env.example` к одному контракту;
- добавление Alembic;
- фиксирование схемы и миграций;
- добавление correlation_id и audit trail;
- pagination для тяжелых списков.

Этап 1. RabbitMQ без ломки текущего execution model
- поднять RabbitMQ в `docker-compose.yml`;
- перевести Celery broker на RabbitMQ;
- Redis оставить как result backend;
- добавить outbox/inbox tables;
- перевести dispatch на transactional publish.

Этап 2. Staging data layer
- ввести staging schema / staging_db;
- extraction workers пишут не на диск как конечную форму, а в staging partitions;
- добавить cleanup staging data.

Этап 3. Normalized data layer
- ввести normalized schema / normalized_db;
- нормализованные данные сохранять в БД как dataset snapshots;
- исходные extracted files удалять после success;
- обучение переключить с файловой normalized model на DB-backed snapshots.

Этап 4. Model registry и versioning
- добавить model_registry и model_version;
- promotion перевести на registry/version;
- связать model version с dataset snapshot, feature schema и code revision.

Этап 5. Масштабирование и эксплуатация
- tuning очередей, worker pools, DB partitioning;
- cleanup policies;
- quotas;
- расширение supported adapters;
- optional вынос staging/normalized DB на отдельные инстансы.

15. Почему это решение лучше альтернатив

Лучше, чем сразу полностью переписывать систему:
- потому что сохраняет текущие API, текущие сервисы и текущие UI-потоки;
- потому что вводит отказоустойчивость по слоям;
- потому что минимизирует миграционный риск;
- потому что позволяет внедрять RabbitMQ и DB-backed processing постепенно.

Лучше, чем оставаться на файловом пайплайне:
- потому что файлы плохо подходят для управления жизненным циклом 200 ГБ данных;
- потому что staging/normalized layers дают контроль, чистку и lineage;
- потому что это снижает memory pressure и упрощает recovery.

Лучше, чем сразу заводить слишком много новых технологий:
- потому что на первом этапе можно сохранить текущий PostgreSQL cluster;
- потому что Redis можно не выбрасывать сразу;
- потому что Celery можно оставить;
- потому что это уменьшает риск остановки проекта на "инфраструктурной революции".

16. Моя итоговая рекомендация

Я рекомендую принять следующую целевую стратегию:

1. Не ломать текущую систему.
2. Перевести брокер межсервисного обмена на RabbitMQ.
3. Ввести transactional outbox и idempotent consumers.
4. Ввести staging data layer для данных после разархивации.
5. Ввести normalized data layer как источник истины для обучения.
6. Удалять extracted raw files после успешной записи и нормализации.
7. Добавить ручную очистку staging и normalized данных.
8. Добавить обязательное версионирование моделей через model registry.
9. Перевести эволюцию схемы на Alembic migrations.
10. Привести все настройки к строгому `.env` / `.env.example` контракту.

17. Что я бы делал дальше после утверждения этой концепции

Если эту концепцию утверждать, следующим шагом правильно делать не код сразу, а технический design package:
- детальная схема сервисов;
- перечень новых таблиц и миграций;
- контракт сообщений RabbitMQ;
- новая карта `.env.example`;
- дорожная карта по этапам;
- список изменений по каждому сервису;
- тестовая стратегия и rollout plan.

На текущем этапе реализация не начата.
