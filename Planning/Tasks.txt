ЗАДАЧА 6. УНИВЕРСАЛЬНЫЙ ИНТЕРФЕЙС ДЛЯ JSON, JSONL И NDJSON
==========================================================

Цель
----
Проанализировать файл scripts/json_data.py и изменить его реализацию так, чтобы он предоставлял
простой и универсальный интерфейс для работы с файлами форматов:
- json
- jsonl
- ndjson

Функциональные требования
-------------------------
Скрипт должен уметь:
1. Записывать данные в файл.
2. Читать данные из файла.
3. Обновлять данные в файле.

Входные параметры методов
-------------------------
Методы должны принимать:
- путь к файлу;
- данные для записи или обновления.

Поддерживаемые форматы входных данных
-------------------------------------
1. Один словарь:

   {}

2. Список словарей:

   [{}]

Логика выбора формата
---------------------
Логика работы должна определяться по расширению файла:
- .json
- .jsonl
- .ndjson

Правила для формата .json
-------------------------
1. Если передан словарь {}, сохранить его как обычный JSON-объект.
2. Если передан список словарей [{}], сохранить его как JSON-массив.
3. При чтении файл должен возвращать:
   - JSON-объект;
   - или список JSON-объектов.

Правила для форматов .jsonl и .ndjson
-------------------------------------
1. Каждая запись должна сохраняться отдельной строкой.
2. Если передан словарь {}, записать одну строку.
3. Если передан список словарей [{}], записать каждый объект отдельной строкой.
4. При чтении файл должен возвращать список словарей.

Требования к обновлению данных
------------------------------
1. Существующие данные должны считываться из файла.
2. Новые данные должны добавляться или объединяться с существующими.
3. Итоговый результат должен сохраняться обратно в файл в корректном формате.

Обработка ошибок
----------------
Необходимо предусмотреть обработку следующих ситуаций:
- файл не найден;
- неподдерживаемый формат;
- некорректный JSON;
- неверный тип данных;
- ошибка чтения файла;
- ошибка записи файла.

Требования к качеству реализации
--------------------------------
Реализация должна быть:
- простой;
- понятной;
- рабочей;
- расширяемой для добавления новых форматов в будущем.

Ожидаемый результат
-------------------
Файл scripts/json_data.py должен предоставлять универсальный интерфейс для чтения, записи и
обновления данных в форматах .json, .jsonl и .ndjson.

ЗАДАЧА 9. ДОРАБОТКА СКРИПТА scripts/read_format.py ДЛЯ ЧТЕНИЯ PCAP

Цель
----
Проанализировать файл scripts/read_format.py и доработать реализацию так, чтобы скрипт корректно читал файлы формата .pcap, включая пример:

C:\Users\fmark\PythonProjects\Proposal\datasets\dns\TRAIN\CIC-Bell-DNS-2021\reference_non_exfil\Raw-20240207T040928Z-001\PCAPs\malware.pcap

Скрипт должен возвращать корректные данные по PCAP-файлу и формировать packet records в структурированном виде для дальнейшей обработки.

Контекст
--------
Текущая задача касается базового чтения PCAP и извлечения packet-level данных.

Полное сетевое декодирование DNS/HTTP в рамках этой задачи не требуется. Для DNS, HTTP и других L7-протоколов нужен отдельный протокольный парсер/декодер, поскольку это требует отдельной логики разбора payload, реконструкции TCP-сессий, обработки фрагментации и отдельного набора тестов.

Функциональные требования
-------------------------
Скрипт должен уметь:

1. Определять и корректно читать файлы формата .pcap.

2. Возвращать содержимое PCAP-файла в формате, совместимом с текущей структурой данных, включая поле:

   'content_base64': '1MOyoQIABAAAAAAAAAAAAAAABAABAAAApW6oXWI6BAB****'

   То есть бинарное содержимое .pcap должно корректно кодироваться в base64, без повреждения данных.

3. Извлекать базовые packet records из PCAP-файла.

4. Для каждого packet record возвращать доступные метаданные, например:

   - timestamp;
   - packet length;
   - captured length;
   - link/network protocol, если определяется;
   - source IP, если есть;
   - destination IP, если есть;
   - transport protocol, если есть;
   - source port, если есть;
   - destination port, если есть;
   - payload length или признак наличия payload.

5. Корректно обрабатывать бинарный формат .pcap без попытки читать его как текстовый файл.

6. Возвращать данные в стабильной структурированной форме, пригодной для дальнейшего анализа и последующей интеграции с отдельными протокольными декодерами.

Ожидаемое поведение
-------------------
При передаче .pcap-файла скрипт должен:

1. Открыть файл в бинарном режиме.
2. Корректно считать его содержимое.
3. Сформировать base64-представление исходного содержимого файла.
4. Распознать PCAP-структуру.
5. Извлечь список packet records.
6. Вернуть результат без ошибок декодирования бинарных данных.

Пример ожидаемой структуры результата
-------------------------------------
{
    "format": "pcap",
    "records": [
        {
            "timestamp": "...",
            "packet_length": 123,
            "captured_length": 123,
            "network_protocol": "IPv4",
            "src_ip": "...",
            "dst_ip": "...",
            "transport_protocol": "UDP",
            "src_port": 12345,
            "dst_port": 53,
            "payload_length": 42
        }
    ]
}

Границы задачи
--------------
Входит в задачу:

- анализ текущей реализации scripts/read_format.py;
- добавление поддержки .pcap;
- корректное чтение бинарного содержимого;
- формирование content_base64;
- базовое извлечение packet records;
- обработка ошибок чтения повреждённых или неподдерживаемых PCAP-файлов;
- добавление или обновление тестов для .pcap.

Не входит в задачу:

- полное декодирование DNS;
- полное декодирование HTTP;
- реконструкция TCP-сессий;
- разбор HTTP-методов, URL, заголовков, тел запросов и ответов;
- разбор DNS questions, answers, record types, response codes;
- глубокий анализ application-layer payload;
- поддержка всех edge cases прикладных протоколов.

Технические требования
----------------------
1. Не читать .pcap через обычный текстовый режим.

   Неверно:
   open(path, "r")

   Правильно:
   open(path, "rb")

2. Base64 должен формироваться из исходных байтов файла.

3. Если используется сторонняя библиотека для чтения PCAP, она должна быть явно добавлена в зависимости проекта.

4. Если библиотека недоступна или формат не поддерживается, скрипт должен возвращать понятную ошибку, а не падать с необработанным исключением.

5. Реализация должна быть расширяемой: дальнейшее DNS/HTTP-декодирование должно подключаться отдельным модулем, не ломая базовый слой чтения packet records.

Критерии готовности
-------------------
Задача считается выполненной, если:

1. scripts/read_format.py корректно обрабатывает файл malware.pcap.
2. В результате присутствует поле content_base64 с base64-представлением бинарного содержимого файла.
3. Возвращается список packet records.
4. Для пакетов извлекаются базовые сетевые метаданные, где они доступны.
5. Бинарные данные не повреждаются при чтении.
6. Скрипт не пытается декодировать PCAP как текст.
7. DNS/HTTP не декодируются в рамках этой задачи, но архитектура позволяет добавить отдельный протокольный декодер позже.
